# SEO audit — 2026-05-26

Scope: the Infrastructure cluster (8-part MikroTik RB5009 series + GPON
standalone). Home & Energy posts were out of scope. Diagnostic angle was
on-page + topical structure, no GSC data.

## Baseline (what was already good)

- Every route had `BlogPosting` structured data, per-post title, description,
  canonical URL, OG tags, Twitter card.
- React Router prerendered HTML — search engines see real content, not an
  empty React root.
- Sitemap maintained by hand; image sitemap entries on `/mikrotik-home-network/`
  and `/solar-report/`.
- ADR-disciplined naming (device vs OS vs vendor) keeps keyword choices
  consistent across the cluster.

## Findings → changes made this session

### 1. Sitemap gaps
- The two CHR variant URLs (`/vps-ipv6-cgnat-mikrotik/chr/`,
  `/mikrotik-ipv6-failover-bgp-bfd/chr/`) were registered as routes and
  prerendered, but missing from `sitemap.xml`. VyOS variants were listed —
  asymmetric.
- **Change**: added both CHR variants to `public/sitemap.xml`.

### 2. Schema type mismatch
- All Infrastructure posts were typed `BlogPosting`. They're step-by-step
  recipes — `TechArticle` is the better Schema.org type and signals
  developer-doc intent to Google.
- **Change**: migrated all 13 Infrastructure routes (8 series + 3 variants +
  GPON standalone, with index counted as part of the 8) from `BlogPosting`
  to `TechArticle`. Added `inLanguage: "en"` and a per-post `dependencies`
  string (tools + versions) — a TechArticle-specific signal.

### 3. No series graph in structured data
- Google had no JSON-LD way to know the 8 posts form a single series. It
  could only infer from prose + the landing page index.
- **Change**: every series post now has
  `isPartOf: { @type: CreativeWorkSeries, name, url }`. The index post
  declares the full `CreativeWorkSeries` with `numberOfItems: 8` and
  `hasPart: [...]` listing all 8 members.

### 4. No breadcrumbs
- No `BreadcrumbList` schema anywhere. Variant URLs (`/vyos/`, `/chr/`)
  especially lost the parent–child relationship in SERPs.
- **Change**: added `BreadcrumbList` JSON-LD on every Infrastructure route.
  Parent posts: `Home › RB5009 series › Post`. Variant posts:
  `Home › RB5009 series › Parent › Variant`. GPON (standalone): 2 levels.

### 5. Landing page H1 + `<title>` were brand-only
- Site title was "marfillaster · notes"; H1 was "Long-running experiments,
  written down." Both identity-forward, no keyword signal for the home page —
  the most important on-page SEO surface on the site.
- **Change**:
  - `<title>` → "Home-lab build logs — MikroTik, solar, EV · marfillaster"
  - H1 → "MikroTik RB5009, residential solar, and plug-in EV — home-lab
    build logs from Cavite."
  - Meta description sharpened with concrete keywords (CGNAT, solar, EV,
    Cavite).
  - `og:site_name` stays "marfillaster · notes" — identity preserved for
    social cards.

### 6. Internal-link gaps (under-linked posts)
- DNS post (3/8) linked only to the index — no cross-links to companions.
- UniFi post (8/8) linked only to index + VLAN.
- BFD post (7/8) linked to index + VPS + (one mention of) Route64; missing
  companion VLAN/DNS/Per-VLAN context.
- **Changes**: added "Companion posts in the series" / "Related in the
  series" / "Next in the series" blocks under each post's `## References`,
  with descriptive anchor text explaining *why* each link is relevant.
  Stayed within CONTEXT.md's linking rules (e.g., BFD still doesn't link
  Route64 as a peer — only as contrast).

### 7. No FAQ / featured-snippet microcontent
- Posts didn't surface the question-shaped queries that would let them rank
  for "what is CGNAT", "HE 6in4 CGNAT", "do I need a VPS for IPv6 over
  CGNAT" — all queries the cluster is well-positioned to answer.
- **Change**: added an FAQ section to the index post (between Verification
  and Glossary) with 5 high-intent Q&A pairs, plus `FAQPage` JSON-LD on the
  route. Questions chosen for query volume × cluster fit:
  1. What is CGNAT?
  2. How do I check if I'm behind CGNAT?
  3. Does Hurricane Electric's 6in4 tunnelbroker work behind CGNAT?
  4. Do I need a VPS to get routable IPv6 behind CGNAT?
  5. What's the difference between a routed /48 and /56?

## Not done in this session (deferred)

### High leverage
- **Per-post OG images.** Today every Infrastructure post except the index
  shares one generic `/og.png`. Bespoke OG images per post would lift
  social-share CTR meaningfully — but the cost is design work, not code.
  Suggested minimum: a templated SVG → PNG generator that stamps the post
  title onto a series-branded background, one per post.
- **Title-tag length pass.** A few post titles exceed Google's ~60-char
  truncation point in SERPs, e.g.:
  - `"Trusted, IoT, and Guest VLANs on RouterOS — hybrid trunks, east-west isolation"` (76 ch)
  - `"Encrypted DNS with a stable resolver address on RouterOS — no VLANs, no IPv6 uplink"` (84 ch)
  - `"Running the UniFi controller on the RB5009 — MongoDB + UniFi in RouterOS containers"` (84 ch)
  Recommendation: shorten the part after the em-dash so the keyword-bearing
  head survives truncation.

### Medium leverage
- **GSC + Bing Webmaster verification.** Submit the sitemap, watch for
  index coverage issues, and re-audit on real query data after 30–60 days.
- **Author / E-E-A-T signal.** `Person` schema is present but minimal.
  Adding `sameAs: [github, …]` and a real `/about` page (rather than the
  embedded section on `/`) gives Google more entity context.
- **Per-post FAQs.** The index FAQ covers the cross-cutting questions.
  Path posts could carry their own (e.g., on Route64: "Is Route64 free?
  What's the catch?"; on VPS: "Which VPS providers route a /48?").
- **Topical gaps worth a separate post.** Searches that the cluster
  almost-but-not-quite answers — likely candidates for a dedicated post
  rather than stretching an existing one:
  - "RouterOS WireGuard server (no broker)" — a pure home-WG-server post.
  - "MikroTik BGP basic primer" — entry-point for readers who land on the
    failover post cold.

### Low leverage / skip
- HowTo schema — Google deprecated rich results for HowTo outside cooking.
  Not worth the risk of misleading markup.
- A separate `/faq/` route — search engines prefer FAQs near the content
  they describe, not in a centralized page. The inline approach already
  taken is the right one.
