## Overview
This is the write-up of replacing my Converge-issued Skyworth GN630V ONT
with an ODI `DFP-34X-2C2` GPON ONU SFP stick plugged directly into the
MikroTik RB5009 SFP+ cage.

```ascii
Converge fiber
    |
    |  SC/APC house fiber -> SC/APC to SC/UPC patch cord
    v
ODI DFP-34X-2C2 GPON SFP stick
    |
    v
RB5009 sfp-sfpplus1 -> VLAN 10 -> DHCP client -> LAN
```

The short version: on my line, Converge authenticated the ONU by GPON serial
number only. After a factory reset, internet worked with just the original
ONT's GPON serial number on the stick and VLAN handling set to transparent.
No MAC cloning, model spoofing, software version spoofing, LOID, or PLOAM
password was needed for this specific ONT/stick/line combination.

This was inspired by [R1BNC's video](https://www.youtube.com/watch?v=92X2lnOek7M)
and his [Converge FiberX ONT replacement post](https://r1bnc.com/post/94/).
Huge thanks also to the [Anime4000/RTL960x](https://github.com/Anime4000/RTL960x)
project and community work around these Realtek-based ONU sticks.

## Tested Hardware
| Component | Tested item |
| --- | --- |
| ISP | Converge ICT residential fiber, Philippines |
| Original ONT | Skyworth GN630V |
| Replacement ONU | ODI `DFP-34X-2C2` GPON/XPON SFP stick, RTL960x-based |
| Stick firmware | `M110_sfp_ODI_220923FS.tar`; reports `V1.0-220923` |
| Router | MikroTik RB5009UG+S+ running RouterOS v7 |
| WAN handoff | VLAN 10, DHCP |
| Fiber connector issue | Converge drop is SC/APC; my ODI stick is SC/UPC |

The connector row matters more than anything else here — it is the trap that
kept the stick unused for six months. Full detail, symptoms, and the fix are
in the **Connector Mismatch** section below.

## What Actually Mattered
The minimum working stick configuration was:

```bash title="ODI stick - minimum working config"
flash set GPON_SN <YOUR_GPON_SN>
flash set VLAN_CFG_TYPE 0
flash set VLAN_MANU_MODE 0
reboot
```

For my line, these were not required:

- `ELAN_MAC_ADDR` or `MAC_KEY` cloning
- RouterOS WAN MAC spoofing
- `PON_VENDOR_ID`, `GPON_ONU_MODEL`, `HW_HWVER`, or `HW_SERIAL_NO` spoofing
- software version spoofing
- `OMCI_OLT_MODE` override
- LOID
- PLOAM password

The important lesson is not "everyone only needs three commands." The lesson
is "prove the minimum before piling on spoofing." My original notes had a
full spoof-heavy setup because I relied heavily on R1BNC's guide as my first
path through the problem. Factory reset testing later proved that this
particular Converge/Skyworth/ODI setup only needed the GPON serial and
transparent VLAN behavior.

## Connector Mismatch
This was the first real trap, and the funny reason the stick sat unused for
about six months: the problem was not the GPON serial, PLOAM, LOID, or model
spoofing. It was the UPC/APC connector mismatch.

| Side | Connector |
| --- | --- |
| Converge fiber | SC/APC, usually green |
| ODI `DFP-34X-2C2` | SC/UPC, blue port |

APC has an angled end face. UPC is flat. The connectors may physically fit,
but the optical interface is wrong. In my testing this showed up as weak
or missing receive power and failed registration.

What I observed with the mismatched connector:

- GPON registration only reached O3
- Rx power reported as 0

I did not observe other registration states or intermittent internet with the
mismatched connector, so treat those as separate clues rather than symptoms
proven by this test.

Use a proper SC/APC to SC/UPC patch cord if your SFP stick is UPC — that is
the adapter path I tested. A cleaner option is buying the APC variant of the
ODI stick, commonly listed as `DFP-34X-2C3`. Another possible path is an
SC/UPC male to SC/APC female adapter, but I did not test that one.

## Gather Values
Log in to the original ONT first and write down only what you need. Do not
publish these values; a GPON serial number is enough to collide with or clone
your line on the same PON.

For the Skyworth GN630V I checked:

| Value | Why |
| --- | --- |
| GPON serial number | The identity Converge accepted on my line; printed on the sticker at the back of the ONT |
| WAN VLAN ID | Converge internet bridge was VLAN 10 |

I did not find PLOAM, LOID, or LOID password values in the Skyworth admin UI,
and I did not need them for this setup. Your line may differ, especially if
it is provisioned on different OLT policy.

## Configure Stick
Do the first configuration with the fiber disconnected. You only need
management access to the SFP stick.

First, insert the ODI stick into the RB5009 SFP+ cage.

The ODI stick management IP is typically `192.168.1.1`, with `admin` as the
password. On RouterOS, prepare the SFP+ port and give the SFP interface a
static address in the same `192.168.1.0/24` network first:

```bash title="RouterOS - reach the SFP stick"
/ip address add address=192.168.1.2/24 interface=sfp-sfpplus1
```

The stick is reachable at `192.168.1.1`. RouterOS automatically adds the
connected route for `192.168.1.0/24`, so no manual route is needed just to
reach the management page or SSH service.

The SSH server may require legacy algorithms. My local SSH config entry looks
like this:

```text title="~/.ssh/config"
Host odi-sfp
    HostName 192.168.1.1
    User admin
    KexAlgorithms +diffie-hellman-group1-sha1
    HostKeyAlgorithms +ssh-rsa
    Ciphers +3des-cbc
    PreferredAuthentications password
    PubkeyAuthentication no
```

Before resetting or changing the stick, back up its current flash
configuration. The web UI has a backup option, but from a shell I also like
capturing `flash all` to a local file:

```bash title="Back up the stick config"
ssh odi-sfp 'flash all' > odi-sfp-flash-backup-$(date +%Y%m%d-%H%M%S).txt
```

Then factory reset the config partition so the stick starts from a clean
baseline. **Confirm the config partition's mtd number on your own stick
first** — the index can differ across stick variants and firmware, and
erasing the wrong mtd can brick the module. Check `cat /proc/mtd` and match
the partition labelled for config before running `flash_eraseall`:

```bash title="ODI stick - factory reset"
ssh odi-sfp

cat /proc/mtd                 # verify which mtd is the config partition
flash_eraseall /dev/mtd3      # mtd3 on my stick — confirm yours
reboot
```

The reboot can take up to two minutes. Before running it, open another
terminal and keep a ping running so you know when the stick is reachable
again:

```bash title="Watch the stick come back"
ping 192.168.1.1
```

After the reset, apply or verify the firmware patch I used. My stick is a
Realtek `RTL9601D`-based ODI `DFP-34X-2C2`; do not use this firmware on a
stick with a different chipset.

This patch step was inherited from R1BNC's guide. I am not sure it is
strictly necessary for this Converge line: my final minimum-config test was
done after a factory reset, but the stick still appeared to be running the
patched firmware image. So this proves the setup works on `220923FS`; it does
not prove that stock firmware would fail.

See the upstream [RTL960x DFP-34X-2C2 firmware README](https://github.com/rajkosto/RTL960x/blob/main/Firmware/DFP-34X-2C2/README.md)
for firmware notes and the `220923FS` patch details.

Firmware file:

```text
M110_sfp_ODI_220923FS.tar
```

Upload that `.tar` from the stick's web UI firmware upgrade page, then let
the stick reboot. If the patch is already installed, skip the upload and just
verify it from SSH:

```bash title="Verify 220923FS patch"
cat /etc/version
ls /etc/scripts/fix_speed.sh /etc/scripts/fix_vlan_tag.sh
```

Expected version marker:

```text
V1.0-220923
```

The `220923FS` tar uses the same `V1.0-220923` version string as the base
firmware, so the more useful check is the presence of `fix_speed.sh` and
`fix_vlan_tag.sh`. The README documents optional activation files under
`/etc/config`, but I did not need to enable those toggles for this working
configuration.

Then apply the minimum settings over SSH (`ssh odi-sfp`) — the same three
`flash set` commands from **What Actually Mattered** above (`GPON_SN`,
`VLAN_CFG_TYPE 0`, `VLAN_MANU_MODE 0`, then `reboot`). The same wait applies
to this reboot. Keep the ping running until `192.168.1.1` responds again.

Verify after reboot:

```bash title="ODI stick - verify settings"
flash get GPON_SN
flash get VLAN_CFG_TYPE
flash get VLAN_MANU_MODE
```

`VLAN_CFG_TYPE=0` and `VLAN_MANU_MODE=0` keep the stick from doing its own
VLAN manipulation. VLAN 10 must pass through to RouterOS so the RB5009 can
run the DHCP client on the VLAN interface.

## RouterOS
Put the DHCP client on VLAN 10 over `sfp-sfpplus1`.

```bash title="RouterOS - Converge WAN over GPON SFP"
# Converge internet VLAN observed from the original ONT.
/interface vlan add interface=sfp-sfpplus1 name=converge-wan vlan-id=10

# DHCP from Converge.
/ip dhcp-client add interface=converge-wan disabled=no

# Let defconf firewall/NAT treat this as a WAN.
/interface list member add list=WAN interface=converge-wan
```

## Swap the ONT
Once the stick and RouterOS are configured, move the fiber handoff to the
RB5009:

Connect fiber through the correct APC-to-UPC patch cord.

If you need to roll back, put the fiber back into the Skyworth ONT.

## Verify
Check the GPON state first:

```bash title="ODI stick"
diag gpon get onu-state
```

The target is:

```text
ONU state: Operation State(O5)
```

Then check DHCP on RouterOS:

```bash title="RouterOS"
/ip dhcp-client print
/ping 8.8.8.8
```

Working state for my setup was:

- ODI stick reaches `Operation State(O5)`
- RouterOS DHCP client binds on `converge-wan`
- internet works through the RB5009 SFP+ port

If the stick says O5 but RouterOS DHCP stays `searching`, look at VLAN
handling first. In my testing, O5 with no DHCP was caused by the stick not
passing VLAN 10 transparently.

## Telemetry & Health Monitoring
After the swap, this stick exposes no telemetry to RouterOS at all.

- **No SNMP on the stick.** Stock `V1.0-220923` ships no `snmpd`, no SNMP
  binary, nothing on UDP 161. The `SNMP_SYS_NAME` flash key is dead Realtek-SDK
  vestige — there is no daemon to enable.
- **No SFP DDM via RouterOS.** SFF-8472 byte 92 (diagnostic monitoring type)
  reads `0x00`: the module declares no diagnostics. `/interface ethernet
  monitor sfp-sfpplus1` returns only the static EEPROM ID (vendor, part,
  wavelength) and reports Tx power as `-inf`, which is false. Do not trust
  that path for anything.

So there is no router-side way to see Rx/Tx power, temperature, or — more
importantly — GPON ONU state. RouterOS only knows the SFP link is up.

The data is all there, though. It is just on the stick, reachable read-only
over SSH with no firmware change:

```bash title="ODI stick - DDM-equivalent telemetry, read-only"
diag pon get transceiver temperature    # module °C
diag pon get transceiver rx-power       # downstream Rx dBm
diag pon get transceiver tx-power       # upstream Tx dBm
diag pon get transceiver voltage        # supply V
diag pon get transceiver bias-current   # laser bias mA
diag gpon get onu-state                 # O1-O7 — the signal RouterOS can't see
```

These return clean human-readable floats — no SFF decode needed — and include
temperature, voltage, and bias, which OMCI does not carry. `omcicli mib get
263` (ANI-G `OpticalSignalLevel`/`TranOpticLevel`, signed int16, `dBm =
value/500`) is a useful secondary cross-check but nothing more.

Live sample from my line, 2026-05-19:

```text
temperature  54.9 C
Rx Power     -21.49 dBm
Tx Power     ~2.0 dBm
Voltage      3.22 V
Bias Current ~20.6 mA
ONU state    Operation State(O5)
```

The OMCI ANI-G decode (`0xd609` → −21.5 dBm) matched the `diag` Rx within
dither, which is what confirmed the SFP-DDM `-inf` Tx reading is an artifact,
not reality.

### The RB5009 is the poller
The stick runs Dropbear 0.48 in inetd mode, advertising only the `password`
auth method. RouterOS's `/system ssh-exec` accepts a `password=` parameter and
negotiates the stick's legacy crypto fine, so the RB5009 polls the stick
itself. No extra host, no
sidecar, no metrics stack.

The requirement was alerting plus a recent on-disk trail, not trend graphs, so
the implementation is a `/system scheduler` running every 2 minutes. One chained
`ssh-exec` opens a single SSH session, runs `onu-state` plus all five
transceiver metrics, and the script parses the output into exactly one log
line per poll:

```text
GPON OK | state=Operation State(O5) temp=54.9 rx=-21.49 tx=2.16 v=3.21 bias=20.5
```

`:log info` while healthy (O5), `:log error` while not O5 / empty response /
stick unreachable. Both carry the metric snapshot, so the Rx-degradation trail
that precedes an APC/UPC outage is in the log either way.

Logging goes to the USB disk, not router NAND — one rotating file capped at
1440 lines (≈48h at 2-minute polls). RouterOS logging rules filter by message regex
(not just topic), so `regex="GPON"` keeps the file GPON-only even though other
scripts log under the same `topics=script`.

The full watcher, verified on-device 2026-05-19:

```bash title="gpon-health-check.rsc"
/system logging action
add name=gponusb target=disk disk-file-name="usb1-part2/gponweb/gpon" \
  disk-lines-per-file=1440 disk-file-count=1 disk-stop-on-full=no

/system logging
add topics=script regex="GPON" action=gponusb

/system script
add name=gpon-health-check policy=read,write,test,password,sensitive source={
  :local host "192.168.1.1"
  :local usr "admin"
  :local pwd "<STICK_ADMIN_PASSWORD>"
  :local cmd "diag gpon get onu-state; diag pon get transceiver temperature; diag pon get transceiver rx-power; diag pon get transceiver tx-power; diag pon get transceiver voltage; diag pon get transceiver bias-current"
  :local getv do={
    :local s $1
    :local k $2
    :local e $3
    :local p [:find $s $k]
    :if ([:typeof $p] = "nil") do={ :return "?" }
    :local r [:pick $s ($p + [:len $k]) [:len $s]]
    :local q [:find $r $e]
    :if ([:typeof $q] = "nil") do={ :return $r }
    :return [:pick $r 0 $q]
  }
  :do {
    :local res [/system ssh-exec address=$host user=$usr password=$pwd \
      command=$cmd as-value]
    :local out ($res->"output")
    :if ([:len $out] = 0) do={
      :log error ("GPON UNHEALTHY | empty response from stick " . $host)
    } else={
      :local state ([$getv $out "ONU state: " ")"] . ")")
      :local temp  [$getv $out "Temperature: " " C"]
      :local rx    [$getv $out "Rx Power: " "  dBm"]
      :local tx    [$getv $out "Tx Power: " "  dBm"]
      :local volt  [$getv $out "Voltage: " " V"]
      :local bias  [$getv $out "Bias Current: " " mA"]
      :local line ("state=" . $state . " temp=" . $temp . " rx=" . $rx . \
        " tx=" . $tx . " v=" . $volt . " bias=" . $bias)
      :if ([:typeof [:find $out "(O5)"]] = "nil") do={
        :log error ("GPON UNHEALTHY | " . $line)
      } else={
        :log info ("GPON OK | " . $line)
      }
    }
  } on-error={
    :log error ("GPON UNHEALTHY | SSH poll failed, stick " . $host . " unreachable")
  }
}

/system scheduler
add name=gpon-health-check interval=2m start-time=startup \
  on-event=gpon-health-check \
  policy=read,write,test,password,sensitive \
  comment="Poll ODI GPON stick; 1-line onu-state+metrics to USB every 2m"
```

Caveats worth stating plainly:

- The stick's admin password sits in clear text in the script. RouterOS has no
  secret store for `ssh-exec`; the mitigation is that router config is
  admin-only.
- A multi-line `source={}` block cannot be pasted over a non-interactive SSH
  session — the console parses it line-by-line and mangles the braces. SCP the
  `.rsc` and `/import` it.
- Temperature, Vcc, and bias have no other safe path. The only lower-level
  source is the BOSA I²C, and probing that on a live PON is exactly the kind of
  thing the TX-calibration safety rule warns against. Don't.
- If full historical trend graphs are ever wanted, the same `diag pon get
  transceiver` data feeds a turnkey Prometheus exporter
  ([Strykar/GPON](https://github.com/Strykar/GPON): exporter + systemd +
  Grafana) on a Linux host with `sshpass`. RouterOS SNMP cannot republish
  these — fixed MIBs only, no custom OID or `extend`. The next section is the
  no-extra-host middle ground: a rolling 48h dashboard, not a metrics stack.

### A dashboard on the router itself
The log answers two questions — "is the line healthy now" and "what did Rx
do in the hours before it died" — but `tail`-ing a 1440-line file over SSH is
not how you want to glance at either. So the same data gets served as one page,
from the router, with no second host: exactly the poller's philosophy applied
to the read side.

The watcher already drops one parseable line per poll to the USB disk. Mount
that file plus a single static `index.html` read-only into a RouterOS
container; the page fetches the `.txt` every poll and draws the Rx / Tx /
temperature sparklines client-side. No Prometheus, no Grafana, no agent —
~260 lines of dependency-free HTML over a file the watcher was already
writing.

The container, veth, and `mountlists` mechanics are exactly those in the
[UniFi-controller-on-the-router post](/unifi-controller-routeros-containers-mikrotik/)
and are not repeated here. Three things are specific to this dashboard:

- **A custom dual-stack server, by design.** The public-v6 goal rules out the
  obvious base image up front: `lipanski/docker-static-website` is busybox
  `httpd` compiled without IPv6, so binding it only ever yields
  `httpd: bad address '[::]:3000'` and LAN v4 at best — never the Route64 v6
  WAN. The dashboard ships its own ~90-line stdlib-only Go static server
  instead: dual-stack `:3000` (Linux `bindv6only=0`,
  one socket serves both the LAN v4 and the GUA), GET/HEAD only, an
  `.html`/`.txt` extension whitelist, explicit `open`+`io.Copy` rather than
  `http.FileServer` (whose `/index.html` → `/` canonical redirect loops), and
  `Cache-Control: no-store` so the per-poll log is never cached.
- **A mounted binary can't be made executable on RouterOS** — there is no
  `chmod`, and a bind mount carries the host's mode bits, not `0755`. The
  server therefore has to ship *inside an image layer* (which preserves mode),
  not via the mount. The Dockerfile is `FROM lipanski/docker-static-website`
  (a layout RouterOS already imports cleanly) with the Go binary `COPY`-ed in
  and the entrypoint swapped to it — the busybox base is kept only as a
  known-good layer wrapper, none of its httpd is used.
- **`/container/add file=` only accepts the legacy uncompressed docker-save
  layout.** A modern Docker/OCI gzip archive is rejected at import
  (`failed to load next entry` / `layer missing, need repull`). The artifact
  that imported came from `docker save` run under colima with the *classic*
  image store (`containerd-snapshotter: false`, overlay2 driver) →
  uncompressed POSIX-tar layers plus `manifest.json`. The UniFi post sidesteps
  this entirely with `remote-image=`; a custom image has no registry, so
  `file=` — and this build-host quirk — is unavoidable.

The on-device config — the veth carries both the LAN v4
and the Route64 GUA, the mount is read-only, and the image is the docker-save
tarball on the USB disk:

```bash title="RB5009 - GPON dashboard container"
/interface/veth/set [find name=veth3-gpon] \
  address=192.168.88.4/24,2001:db8:1234:1::4/64 \
  gateway=192.168.88.1 gateway6=2001:db8:1234:1::1

/container/mounts/add list=gpon-mounts \
  src=/usb1-part2/gponweb dst=/home/static read-only=yes

/container/add file=usb1-part2/gponserve-image.tar interface=veth3-gpon \
  mountlists=gpon-mounts hostname=gponv6 workdir=/home/static \
  start-on-boot=yes logging=yes
```

The mount source `usb1-part2/gponweb` is the *same directory the watcher logs
into*: the dashboard's `index.html` sits next to `gpon.0.txt`, and the page
just fetches its own neighbour. Read-only, so a compromised page process
cannot touch the trail it is displaying.

#### Reaching it over public IPv6
**Route64 delivers unsolicited inbound** — a free tunnel broker is not
outbound-only here. One `forward` accept does it, and exactly one thing about
it matters: it has to sit **before** the `defconf: drop everything else not
coming from LAN` rule.
`wg-route64` is not in the `LAN` interface-list — by design, it is a WAN — so
that defconf drop will discard the inbound SYN unless an explicit accept for
this one destination and port is reached first. Placed ahead of it, the
path just works. (How the Route64 /56 itself gets onto the router is the
[no-VPS IPv6 post](/route64-ipv6-cgnat-mikrotik/); this is just one port
opened on it.)

```bash title="RB5009 - the public-v6 accept, placed before the !LAN drop"
/ipv6/firewall/filter/add chain=forward action=accept \
  connection-state=new protocol=tcp \
  in-interface=wg-route64 \
  dst-address=2001:db8:1234:1::4 dst-port=3000 \
  comment="GPON dashboard: public v6 in" \
  place-before=[find chain=forward \
    comment="defconf: drop everything else not coming from LAN"]
```

From a VPS on neither this LAN nor any Route64 infrastructure, `ping6` to the
GUA returned 3/3 and `curl -6 http://[2001:db8:1234:1::4]:3000/` returned
`HTTP 200` with the full page, and the rule's own packet counter ticked from
zero in step with the requests. That counter stays *low* by design — a `connection-state=new`
accept behind the conntrack established/related accept only ever counts the
opening SYN of each flow while the 9.6 KB response rides the fast path. An
ICMP "it pings" test never touches this rule (it hits the default ICMP
accept); a real TCP fetch is the only thing that proves reachability, which
is why the proof above is a `curl`, not a `ping`.

Caveats, stated as plainly as the poller's:

- **Plain HTTP, no auth, `<meta name="robots" content="noindex">`.** This is a
  read-only personal dashboard on a routable address; treat the URL itself as
  the only thing you don't publish. The hardening that *is* there is the
  server, not transport: GET/HEAD only, extension-whitelisted, `path.Clean`
  traversal-stripped, and living in the same read-only mount it serves so no
  stray file is downloadable. If it needs to be private, reach it over the
  WireGuard side or front it with an authenticating reverse proxy — do not
  bolt TLS onto a 90-line server.
- **The `file=` tarball is build-host-specific.** It is not the image you
  `docker build` locally with a modern toolchain; it is the colima/classic-
  store `docker save` output. The rebuild/deploy recipe lives in the
  Dockerfile header comment for exactly this reason.
