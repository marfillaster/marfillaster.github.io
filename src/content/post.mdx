## Abstract
This is a journal of building a small, opinionated home network on a single
MikroTik RB5009 behind a residential fiber connection that uses carrier-grade
NAT. The ISP gives a dynamic IPv4 address with no inbound reachability and no
usable IPv6. The goals are pedestrian: keep IoT devices and guests away from
the trusted LAN and have real IPv6 so the network stays useful as the rest of
the internet finishes the v6 migration.

The build proceeds in two layers, in the order they were actually installed:
first the LAN is split into VLANs, then IPv6 reachability is recovered through
a $3/month VPS (1&nbsp;TB monthly transfer included) that routes a /48 down to
the house. The first layer — VLAN segmentation, paste-ready and in full — is
its own companion post:
[Trusted, IoT, and Guest VLANs on a MikroTik RB5009](/mikrotik-vlan-guest-iot/);
this log assumes it has been applied and builds the IPv6 layer on top.
Encrypted DNS and a
stable resolver address — the part of this build that needs neither layer —
is broken out, paste-ready and in full, as its own companion post:
[Encrypted DNS with a stable resolver address on RouterOS](/encrypted-dns-stable-resolver-mikrotik/).
None of these steps depend on a specific brand of fiber service; they only
assume that outbound UDP works and that the router can carry a small
WireGuard tunnel. Running the UniFi controller on this same router — so there
is no second always-on box — is its own companion post:
[Running the UniFi controller on the router itself](/unifi-controller-routeros-containers-mikrotik/).

Each numbered section below is paste-ready against a defconf RouterOS&nbsp;v7
setup. The italicized notes after each step are the rationale — what trade-off
was being made and why. The lone appendix documents cost; sub-second IPv6
failover (BGP&nbsp;+&nbsp;BFD) is its own companion post:
[Sub-second IPv6 failover on RouterOS](/mikrotik-ipv6-failover-bgp-bfd/).
The references point at the original gists.

## Design decisions

One protocol choice in this build is load-bearing enough that a reader
substituting parts will want a sentence of justification.

**WireGuard** carries the IPv6 transport because residential CGNAT only
guarantees outbound UDP, and WireGuard needs nothing more than that. Both
endpoints (RouterOS&nbsp;v7 and Linux) speak it in-kernel with a single
config primitive on each side; there is no daemon, no negotiation, and no
MTU two-stepping with an underlying IPsec SA. IPsec/IKEv2, GRE+IPsec, or
OpenVPN would all work; each adds operational surface this build does not
need to pay for.

The static `::/0` route here detects a dead tunnel in ~30&nbsp;seconds. Cutting
that to ~600&nbsp;ms with a BGP route on a BFD-monitored session — why those
two protocols, and the bird2/RouterOS config — is its own optional companion
post: [Sub-second IPv6 failover on RouterOS](/mikrotik-ipv6-failover-bgp-bfd/).

## 1. Topology and address plan
```ascii
                       Internet (IPv4 + IPv6)
                              │
                ┌─────────────┴─────────────┐
                │  VPS — Ubuntu, routed /48 │
                │  <VPS_IP>                 │
                │  enp3s0: provider GUA     │
                │  wg0:    <LAN_PREFIX>:0::1│
                └─────────────┬─────────────┘
                              │ WireGuard / UDP 51820
                              │ (only IPv6 transits the tunnel)
                ┌─────────────┴─────────────┐
                │   Converge fiber + CGNAT  │
                │   (IPv4 outbound only)    │
                └─────────────┬─────────────┘
                              │
                ┌─────────────┴─────────────┐
                │  MikroTik RB5009 — edge   │
                │  • DHCPv4 + RA + RDNSS    │
                │  • DoH resolver           │
                │  • WireGuard client       │
                └──┬──────────┬──────────┬──┘
                   │          │          │
        ether2/3 (trunks)  ether4/5 (LAN-only access)
                   │
        ┌──────────┴──────────┐
        │  UniFi 6 APs ×2     │
        │  untagged = mgmt    │
        │  tag 10 = IoT SSID  │
        │  tag 20 = Guest SSID│
        └──────────┬──────────┘
                   │
       LAN 1   IoT 10   Guest 20
   192.168.88/24  .89/24  .90/24
   <LAN_PREFIX>:1::/64  :10::/64  :20::/64
```

One box does everything the LAN side needs. The RB5009 terminates the WAN,
runs DHCPv4 and IPv6 RA, runs DoH outbound, and hosts the WireGuard client
toward the VPS. Two UniFi 6 APs hang off bridge ports configured as hybrid
trunks: untagged
frames carry AP management on the main LAN, tagged frames carry IoT and Guest
SSID traffic.

<Rationale>
One always-on device is enough. The UniFi controller also runs on this same
RB5009 as RouterOS containers — no second box, no second attack surface —
which is its own companion post:
[Running the UniFi controller on the router itself](/unifi-controller-routeros-containers-mikrotik/).
</Rationale>

#### Address plan

| VLAN    | Tagging      | Role                | IPv4              | IPv6                  |
| ------- | ------------ | ------------------- | ----------------- | --------------------- |
| VLAN 1  | untagged     | main LAN, AP mgmt   | `192.168.88.0/24` | `<LAN_PREFIX>:1::/64` |
| VLAN 10 | tagged       | IoT SSID            | `192.168.89.0/24` | `<LAN_PREFIX>:10::/64`|
| VLAN 20 | tagged       | Guest SSID          | `192.168.90.0/24` | `<LAN_PREFIX>:20::/64`|
| —       | WG transport | VPS ↔ MikroTik      | n/a               | `<LAN_PREFIX>:0::/64` |

Each VLAN is a separate L3 boundary at the router. The IoT and Guest VLANs
deliberately reuse the same slice IDs as their VLAN numbers in the IPv6 plan
(`:10::/64`, `:20::/64`) so the mapping stays obvious in tcpdump.

## 2. Conventions and placeholders
Every snippet below uses angle-bracketed placeholders. Substitute them before
pasting; the rest of the text is literal RouterOS or bash.

| Placeholder                   | Meaning                                                                                          |
| ----------------------------- | ------------------------------------------------------------------------------------------------ |
| `<LAN_PREFIX>`                | Routed IPv6 /48 (or /56) the VPS hands you. Drop the trailing zeros, e.g. `2001:db8`.            |
| `<ULA_PREFIX>`                | Locally-generated ULA. `python3 -c 'import secrets; print(f"fd{secrets.token_hex(5)}")'`.        |
| `<VPS_IP>`                    | VPS public IPv4 from the provider panel.                                                         |
| `<VPS_NIC>`                   | VPS public interface. `ip -o link` shows it (typical: `enp3s0`, `ens3`).                         |
| `<VPS_PUBKEY> / <MT_PUBKEY>`  | WireGuard public keys, one per side. Each is printed during section&nbsp;4.                      |

Documentation prefixes from RFC&nbsp;3849 (`2001:db8::/32`) and RFC&nbsp;4193
(`fd00::/8`) work as stand-ins while testing. Snippets assume the defconf
bridge name `bridge` and ports `ether2`–`ether5`; rename the interfaces where
they appear.

## 3. LAN segmentation comes first
The network is split before any services are added on top of it: main LAN on
VLAN&nbsp;1 untagged, IoT and Guest on tagged VLANs 10 and 20 sharing the AP
uplinks, each VLAN its own L3 boundary with per-VLAN DHCP and east-west
isolation enforced in the `forward` chain.

This first layer needs nothing else in this build — no IPv6, no VPS, no
WireGuard — so it is broken out, paste-ready and in full, as its own
companion post:
[Trusted, IoT, and Guest VLANs on a MikroTik RB5009](/mikrotik-vlan-guest-iot/).
That post covers the bridge VLAN table and L3 gateways, the per-VLAN DHCP
scopes, and the input/forward firewall, with the hybrid-trunk rationale.

**Everything below assumes it has been applied** — specifically the
`vlan-iot` and `vlan-guest` interfaces and the bridge VLAN table it creates.
The IPv6 layer simply adds a global and ULA prefix to those same interfaces.

<Rationale>
Hybrid trunks instead of a tagged management VLAN keep AP onboarding trivial:
APs boot on the main LAN, only client SSID traffic is tagged. Isolation lives
in `forward`, not by starving clients of DHCP/DNS in `input`. The companion
post is where that trade-off is made concrete.
</Rationale>

## 4. IPv6 over WireGuard via a routed /48
CGNAT eats inbound IPv4. IPv6 sidesteps that problem entirely — but only if
the network has real, routable IPv6. The recipe is a $3/month VPS that routes
a /48 to its instance, a WireGuard tunnel from the RB5009 to that VPS, and
per-VLAN /64s carved out of the /48.

<Rationale>
A routed /48 is what unlocks the address plan in section&nbsp;1. With an
on-link /64 the only options are NDP-proxying and a single subnet; with a
routed /48 there are no helper daemons and 65k subnets to spare.
</Rationale>

The transport is WireGuard to a VPS you control. Residential CGNAT
guarantees only outbound UDP, which is all WireGuard needs, and terminating
on your own instance makes the routed /48, the return routing
(section&nbsp;4.1), and the failover ([its own companion
post](/mikrotik-ipv6-failover-bgp-bfd/)) yours to define rather than a third
party's. The only recurring cost is one cheap VPS
(Appendix&nbsp;A); everything reachable is on an endpoint you own, with no
dependence on a broker's PoP, ToS, or portal.

If you would rather not pay for a VPS and can accept a single *free* IPv6
uplink that fails fast to IPv4 when it drops, the companion post —
[Routed IPv6 for a segmented IPv4-only LAN behind CGNAT](/route64-ipv6-cgnat-mikrotik/) —
does exactly that with Route64's free WireGuard /56 and no VPS.

### 4.1 Return routing: why each /64 has to appear in AllowedIPs
The provider routes the /48 to the VPS, then configures one address from it on
the public NIC at /48 mask. That installs a connected route for the entire
/48 on the public interface, which competes with the WireGuard tunnel for
return traffic. Listing the parent /48 in `AllowedIPs` ties on prefix length
and loses to the public NIC by metric. Listing each LAN /64 separately wins
by longest-prefix match.

```ascii
# Routing table on the VPS — what makes return traffic work
<LAN_PREFIX>::/48        dev <VPS_NIC>  metric 256   ← provider's connected route (harmless)
<LAN_PREFIX>:1::/64      dev wg0                    ← from AllowedIPs, wins by /64 > /48
<LAN_PREFIX>:10::/64     dev wg0                    ← from AllowedIPs
<LAN_PREFIX>:20::/64     dev wg0                    ← from AllowedIPs

# Listing the parent /48 in AllowedIPs would lose to the connected route
# on the public NIC and return traffic would disappear. List each /64.
```

<Rationale>
This is the one detail that silently breaks the whole setup. Egress works,
ping works, then the first packet to a SLAAC client falls off the public NIC
and the network looks broken in a way that no log line explains. The fix is
paste-only — but only if it is paste-correct.
</Rationale>

### 4.2 VPS — WireGuard relay
```bash title="VPS wg0 + AllowedIPs per /64"
set -e
apt-get update -qq
apt-get install -y -qq wireguard

cat >/etc/sysctl.d/99-wg-relay.conf <<'EOF'
net.ipv6.conf.all.forwarding = 1
net.ipv6.conf.default.forwarding = 1
net.ipv6.conf.<VPS_NIC>.accept_ra = 2
EOF
sysctl --system >/dev/null

umask 077
mkdir -p /etc/wireguard
wg genkey | tee /etc/wireguard/server.key | wg pubkey > /etc/wireguard/server.pub
VPS_PRIVKEY=$(cat /etc/wireguard/server.key)

cat >/etc/wireguard/wg0.conf <<EOF
[Interface]
PrivateKey = ${VPS_PRIVKEY}
Address    = <LAN_PREFIX>:0::1/64
ListenPort = 51820
MTU        = 1420

[Peer]
PublicKey  = <MT_PUBKEY>
# One entry per LAN /64. Longest-prefix match beats the connected
# /48 on the public NIC and keeps return traffic on wg0.
AllowedIPs = <LAN_PREFIX>:0::2/128, <LAN_PREFIX>:1::/64, <LAN_PREFIX>:10::/64, <LAN_PREFIX>:20::/64
PersistentKeepalive = 25
EOF

ufw allow 51820/udp comment "WireGuard"
ufw route allow in on wg0
ufw route allow out on wg0
ufw reload || true
systemctl enable --now wg-quick@wg0
echo "VPS public key: $(cat /etc/wireguard/server.pub)"
```

### 4.3 MikroTik — WireGuard client and main-LAN IPv6
The static `::/0` with `check-gateway=ping` detects a dead tunnel in roughly
30 seconds. The companion post
[Sub-second IPv6 failover on RouterOS](/mikrotik-ipv6-failover-bgp-bfd/)
replaces it with BGP+BFD for ~600&nbsp;ms detection if that matters.

```bash title="MikroTik WireGuard + main LAN IPv6"
/interface/wireguard add name=wg-host listen-port=51820 mtu=1420
/interface/wireguard/peers add interface=wg-host name=vps \
    public-key="<VPS_PUBKEY>" \
    endpoint-address=<VPS_IP> endpoint-port=51820 \
    allowed-address=::/0 \
    persistent-keepalive=25s

/ipv6/address add address=<LAN_PREFIX>:0::2/64 interface=wg-host advertise=no
/ipv6/address add address=<LAN_PREFIX>:1::1/64 interface=bridge advertise=yes
/ipv6/address add address=<ULA_PREFIX>:1::1/64 interface=bridge advertise=yes comment="LAN ULA"

# Static default with ping liveness — detection ~30 s.
# The BGP+BFD failover companion post replaces this for ~600 ms. The
# comment is what its `/ipv6/route/remove` matches against, so set it now.
/ipv6/route add dst-address=::/0 gateway=<LAN_PREFIX>:0::1%wg-host \
    check-gateway=ping comment="vps primary"

/ipv6/firewall/filter add chain=input action=accept in-interface=wg-host \
    comment="accept input from VPS WG peer" \
    place-before=[find where chain=input and comment="defconf: drop everything else not coming from LAN"]
```

### 4.4 Per-VLAN addresses and RA RDNSS
Each VLAN gets a GUA from the /48 and a stable ULA from the locally-generated
`fd…` prefix. The ULA is what gets advertised as the DNS server — it stays
the same when the global prefix is renumbered, which it eventually will be.

```bash title="GUA + ULA + RDNSS per VLAN"
/ipv6/address
add interface=vlan-iot   address=<LAN_PREFIX>:10::1/64 advertise=yes comment="IOT GUA"
add interface=vlan-iot   address=<ULA_PREFIX>:10::1/64 advertise=yes comment="IOT ULA"
add interface=vlan-guest address=<LAN_PREFIX>:20::1/64 advertise=yes comment="GUEST GUA"
add interface=vlan-guest address=<ULA_PREFIX>:20::1/64 advertise=yes comment="GUEST ULA"

/ipv6/nd
add interface=bridge     advertise-dns=yes dns=<ULA_PREFIX>:1::1  managed-address-configuration=no other-configuration=no
add interface=vlan-iot   advertise-dns=yes dns=<ULA_PREFIX>:10::1 managed-address-configuration=no other-configuration=no
add interface=vlan-guest advertise-dns=yes dns=<ULA_PREFIX>:20::1 managed-address-configuration=no other-configuration=no
```

### 4.5 IPv6 firewall and anti-spoofing
Mirror the IPv4 isolation in IPv6, then drop traffic whose source prefix does
not belong on the ingress VLAN. SLAAC makes prefix forgery trivially cheap;
the address-list filters make it not work.

```bash title="IPv6 isolation + anti-spoof"
/ipv6/firewall/filter
add chain=input action=accept in-interface=vlan-iot   protocol=udp dst-port=547 comment="IOT: DHCPv6"
add chain=input action=accept in-interface=vlan-iot   protocol=udp dst-port=53  comment="IOT: DNSv6"
add chain=input action=accept in-interface=vlan-guest protocol=udp dst-port=547 comment="GUEST: DHCPv6"
add chain=input action=accept in-interface=vlan-guest protocol=udp dst-port=53  comment="GUEST: DNSv6"

add chain=forward action=drop in-interface=vlan-iot   out-interface=bridge   connection-state=new comment="IOT !-> LAN (v6)"
add chain=forward action=drop in-interface=vlan-guest out-interface=bridge   connection-state=new comment="GUEST !-> LAN (v6)"
add chain=forward action=drop in-interface=vlan-guest out-interface=vlan-iot connection-state=new comment="GUEST !-> IOT (v6)"

/ipv6/firewall/address-list
add list=lan-legit   address=<LAN_PREFIX>:1::/64
add list=lan-legit   address=<ULA_PREFIX>:1::/64
add list=lan-legit   address=fe80::/10
add list=iot-legit   address=<LAN_PREFIX>:10::/64
add list=iot-legit   address=<ULA_PREFIX>:10::/64
add list=iot-legit   address=fe80::/10
add list=guest-legit address=<LAN_PREFIX>:20::/64
add list=guest-legit address=<ULA_PREFIX>:20::/64
add list=guest-legit address=fe80::/10

/ipv6/firewall/filter
add chain=forward action=drop in-interface=bridge     src-address-list=!lan-legit   comment="LAN: anti-spoof"
add chain=forward action=drop in-interface=vlan-iot   src-address-list=!iot-legit   comment="IOT: anti-spoof"
add chain=forward action=drop in-interface=vlan-guest src-address-list=!guest-legit comment="GUEST: anti-spoof"
```

## 5. Encrypted DNS with stable resolver addresses
Upstream DNS leaves the house through Cloudflare DoH; downstream, the router
is the resolver and advertises itself via RA RDNSS at a ULA, so SLAAC prefix
churn never moves the DNS server clients have memorized. Every DHCPv4 scope
is set `dns-none=yes` — resolution is uniformly the ND RDNSS path.

This piece stands entirely on its own — it needs neither the VLAN
segmentation of §3 nor the routed IPv6 of §4 (the ULA is locally assigned;
DoH egresses over IPv4) — so it is written up, paste-ready and in full, as
its own companion post:
[Encrypted DNS with a stable resolver address on RouterOS](/encrypted-dns-stable-resolver-mikrotik/).
That post covers the DoH bootstrap pins, the LAN ULA and `router.lan`
record, `advertise-dns=self`, the uniform `dns-none=yes` posture, and the
verification — the same configuration this build runs.

<Rationale>
Pinning the DoH bootstrap A-only matters more here than in the standalone
post: a client's first query must not ride the IPv6 path, which only becomes
reachable after the WireGuard tunnel handshakes (and, with the BGP+BFD
failover companion, after BFD converges). The DNS companion post explains
the rest.
</Rationale>

## 7. Update — keeping streaming off the metered tunnel
*Added May 16, 2026. Running the build surfaced a failure mode the original
design did not anticipate.*

Two problems compounded on the IPv6 leg, both rooted in the same fact: every
global IPv6 packet from the house egresses the VPS, so to the outside world it
originates from a datacenter ASN, not the residential ISP.

- **VPN / geo detection.** Streaming apps increasingly treat datacenter
  address space as a VPN and either refuse to play or pin the wrong region.
  Because Happy&nbsp;Eyeballs (RFC&nbsp;8305) prefers IPv6, the app reaches for
  the flagged path *first*; the perfectly good native IPv4 route never gets a
  turn.
- **Metered bandwidth.** The VPS plan is 1&nbsp;TB/month with overage billed
  per Appendix&nbsp;A. High-bitrate video has no reason to transit a rented
  tunnel — it just burns the cap.

The realization: not every trusted device needs global IPv6. The cheapest and
most robust lever is a VLAN that has a **ULA but no GUA**. With no global
address there is no global v6 route at all — the client uses native ISP IPv4
for anything internet-facing, while the ULA still carries router DNS (RA
RDNSS, §4.4) and on-link v6. This is failsafe by construction: it is the
*absence of a route*, not a firewall rule something could slip past, and the
per-VLAN anti-spoof list (§4.5) enforces that nothing forges its way back onto
a global prefix.

*Why not just drop the GUA's forward path in the firewall instead? Because the
client would still autoconfigure a GUA, still prefer it per Happy&nbsp;Eyeballs,
and stall for the fallback timeout on every new connection. Removing the GUA
means the client never attempts the tunnel — there is nothing to fall back
from.*

### 7.1 A ULA-only trusted VLAN
VLAN&nbsp;30 is a normal trusted VLAN — `LAN` interface-list membership is the
only thing "trusted" means here — with exactly one difference from
VLAN&nbsp;1: no GUA is assigned, so there is no global v6 route to leave on.

```bash title="VLAN 30 — trusted, ULA-only"
/interface/vlan add interface=bridge name=vlan30 vlan-id=30
/interface/bridge/vlan add bridge=bridge vlan-ids=30 \
  tagged=bridge,ether2,ether3 comment="VLAN30 trusted, tagged to APs"

/ip/address add address=192.168.91.1/24 interface=vlan30
/ip/pool add name=vlan30-pool ranges=192.168.91.100-192.168.91.200
/ip/dhcp-server add name=vlan30-dhcp interface=vlan30 address-pool=vlan30-pool lease-time=1d
/ip/dhcp-server/network add address=192.168.91.0/24 gateway=192.168.91.1 dns-none=yes

/interface/list/member add list=LAN interface=vlan30

# ULA only — deliberately no <LAN_PREFIX>:30::1 GUA.
/ipv6/address add interface=vlan30 address=<ULA_PREFIX>:30::1/64 advertise=yes comment="VLAN30 ULA"
/ipv6/nd add interface=vlan30 advertise-dns=self \
  managed-address-configuration=no other-configuration=no

/ipv6/firewall/address-list
add list=vlan30-legit address=<ULA_PREFIX>:30::/64
add list=vlan30-legit address=fe80::/10
/ipv6/firewall/filter add chain=forward action=drop in-interface=vlan30 \
  src-address-list=!vlan30-legit comment="VLAN30: anti-spoof"
```

*The DNS lines (`dns-none=yes`, `advertise-dns=self`) are the standard
RDNSS posture from §5 — see the DNS companion post for why. One detail is
VLAN&nbsp;30-specific: its RA omits the `mtu=1420` clamp the tunnel-bound
VLANs carry (§4.4), because VLAN&nbsp;30's v6 never reaches the WireGuard
path and clamping would only shrink local traffic for nothing.*

Move the main client SSID onto VLAN&nbsp;30 in the controller, leaving
VLAN&nbsp;1 for the few devices that genuinely want global v6. Guest
(VLAN&nbsp;20) gets the same treatment — it never needed metered global
reachability, so drop its GUA and the matching `guest-legit` entry from §4.5:

```bash title="Strip the Guest GUA too"
/ipv6/address remove [find comment="GUEST GUA"]
/ipv6/firewall/address-list remove [find list=guest-legit address=<LAN_PREFIX>:20::/64]
```

IoT (VLAN&nbsp;10) keeps its GUA: its handful of outbound v6 flows are
negligible against the cap, and some devices behave better with working v6
than with a half-deprecated one.

### 7.2 The cutover gotcha
A device that is associated at the moment its SSID's VLAN changes keeps the
*old* VLAN's SLAAC addresses until their lifetimes expire — RouterOS
advertises prefixes with a one-week preferred / four-week valid lifetime by
default. So for up to a week those devices still prefer the old GUA, the
anti-spoof rule on the new VLAN drops it, and they fall back to IPv4 with a
brief Happy-Eyeballs delay on the first connection to each host.

This is harmless — nothing leaks; the drop counter on the new anti-spoof rule
is exactly that retried traffic being contained — but it is not instant. There
is no router-side fix, because a prefix can only be retracted on the link the
client has already left. The one effective cleanup is a single Wi-Fi
reconnect (or reboot) of each moved device: on re-association it rebuilds its
address set for the new link and the stale prefix is simply never recreated.

*Net effect: streaming and other high-bitrate traffic from the main SSID now
rides native ISP IPv4 — no datacenter ASN, no VPN flag, no metered GB — while
DNS, on-link IPv6, and the trusted-LAN posture are unchanged. The VPS tunnel
goes back to doing what it is actually for: giving the devices that want it
real, routable IPv6.*

## A. Appendix A — Cost and provider notes
The IPv6 leg is the only recurring cost in the build, and it depends entirely
on whether the VPS provider routes a prefix to the instance or only hands out
an on-link /64. Providers known to route prefixes (verify on your specific
plan): WebHorizon&nbsp;SG (/48), Hetzner Cloud (/64 effectively routed via
NDP-proxy), Linode/Akamai (/64 default, /56 on request), Oracle Cloud (/56).

| Line item            | On-link /64 (e.g. Vultr) | Routed /48 (this build)     |
| -------------------- | ------------------------ | --------------------------- |
| VPS plan             | `$5 / mo`                | `$3 / mo (routed /48, 1 TB)`|
| Reserved IPv6 fee    | `$3 / mo`                | `$0`                        |
| Bandwidth overage    | `$0.01 / GB`             | `$0.0025 / GB`              |
| LAN address space    | `one /64`                | `/48 — 65 k /64s`           |
| Extra daemon on VPS  | `ndppd`                  | `none`                      |
